0、项目架构与运行

0.1.IOC和AOP

IOC

IOC（invert of control） 就是控制反转，是一种思想而不是具体实现，具体来说就是把 bean 的创建交由 Spring 框架进行管理。IOC 容器是 Spring 实现 IOC 的载体，IOC 容器实际上就是一个 Map，里面存放着各种对象。DI（Dependency Injection）依赖注入是 IOC 的一种具体实现。

在我做的项目中 IOC 给我最大的感受就是简单和可拓展性。比如 service 层的下单服务，之后如果对下单服务进行更新和优化，如果没有使用 IOC 就需要将之前 controller 层使用过该服务的代码都需要修改，而如果利用了 IOC 直接新加一个下单服务的 impl，然后再将之前的 impl 的注解掉就行了。

在使用的时候还可以用 @Qualifier("") 显式地将服务写出来，但是不推荐，会使得修改的时候很麻烦。

AOP

AOP（Aspect oriented Programming） 是面向切面编程。项目中在日志模块用到，就是写了一个 LogAspect 类，然后写一个方法加上 @Pointcut("execution(* com.nowcoder.example.service.*.*(..))") 注解声明哪些包下的哪些类哪些方法需要织入，然后再用 @Before @After 等注解写需要去执行的动作。一般都是运行时织入，如果有接口就采用 JDK 动态代理，如果没有接口就采用 CGLIB 动态代理。

0.2.MVC

后端在收到一个请求后会由 DispatcherServlet 统一去处理，然后由 HandlerMapping 会告诉 DispatcherServlet 由哪个 HandlerAdapter 来调用 controller 来处理这个请求，处理完之后会返回给 DispatcherServlet 一个 ModelAndView，然后再由 ViewResolver 处理返回给 DispatcherServlet，最后将处理好的 view 返回给前端。

3.1.下单秒杀

下单的时候锁库存会产生少卖的问题，设置一个时间，超过了这个时间还没支付库存就会回退回去。一般会这样设置，用户体验更好。

付款的时候锁库存会产生超卖的问题，一般不会这样用。

order_info 表的 id 存 20211124000000000000，前面八位存的是年月日，一般时间过长的订单信息会被拆分出去，所以这样设置可以更好地利于我们拆分。

怎么去保证订单号不重复的？

serial_number 表用于记录订单后面部分的流水号码，在查询的时候会在 sql 中加入 for update 将其锁住，防止其它事务去读取，导致订单后面流水号相同，出现问题。（这个表主要是用来存业务的最大序号的）

/order/create 去创建订单，然后去 service 层调用 createOrder 方法，这个方法是加了 @Transactional 注解保证事务的。首先判断购买的数量否合法，用户是否登录了，然后去检验商品是否存在，库存时候充足。都满足之后就会去创建订单，首先会扣减库存，然后去生产订单信息，最后去更新 item 表的销量信息，然后返回给前端。

3.2.数据库的事务与锁

锁相关

粒度

表锁、行锁

类型

共享锁（S)：行锁，允许事务读一行数据；

排他锁（X)：行锁，允许事务删除或更新一行数据；

意向共享锁（IS)：表锁，事务想要获得一张表中某几行的共享锁；

意向排他锁（IX)：表锁，事务想要获得一张表中某几行的排他锁；

机制

无锁：MVCC 对于正在更新的数据，InnoDB 会去读取该行的一个快照数据（undo log）；

加锁：X 锁：SELECT ... FOR UPDATE；S锁：SELECT ... LOCK IN SHARE MODE；

算法

Record Lock：单个行记录上的锁，总是会去锁住索引记录；

Gap Lock：间隙锁，锁定一个范围，但不包含记录本身；

Next-Key Lock：Record Lock + Gap Lock，锁定一个范围，并且锁定记录本身；

死锁

事务1：update ... where id=1; update ... where id=2;

事务2：update ... where id=2; update ... where id=1;

当事务 1 和事务 2 同时锁住 1、2 的时候产生了循环依赖，导致死锁。

解决方案：

  1. 超时回滚：innodb_lock_wait_timeout，当一个等待时间超过设置的某一阈值时，其中一个锁定范围小的回滚，另一个等待的事务就能继续进行；
  2. 死锁检测（常用）：wait-for graph，采用等待图的方式进行死锁检测，以事务为单位画出依赖图，相互依赖就说明会产生死锁。

不同日志作用

redo log

  1. 当事务提交时，必须先将该事务的所有日志写入到 redo log 进行持久化，待事务的 commit 操作完成才算完成；
  2. redo log 是顺序写的，在数据库运行时不需要对该文件进行读取操作；
  3. 每次写入 redo log 文件后，InnoDB 引擎都需要调用一个 fsync 刷盘操作；
  4. redo log 通常是物理日志，存储的是二进制数据，记录的是页的物理修改操作；

redo、bin log区别

  1. redo log 是在存储引擎层产生，bin log 是在数据库的上层产生的，并且 bin log 不仅仅针对 InnoDB 引擎，MySQL 中任何存储引擎对于数据库的更改都会产生二进制日志；
  2. bin log 是一种逻辑日志，其记录的是对应的 SQL 语句，而 InnoDB 存储引擎层面的 redo log 是物理格式日志，其记录的是对于每个页的修改；
  3. bin log 只是在事务提交完成后再进行一次写入，而 redo log 在事务进行中不断地写入，这表现为日志并不是随事务提交的顺序进行写入的。

undo log

  1. undo log 存放在数据库内部的一个特殊段（segment）中，这个段称为 undo 段（undo segment）；
  2. undo log 是逻辑日志，因此只是将数据库逻辑地恢复到原来的样子（执行 SQL）；
  3. undo log 的另一个作用是 MVCC，即在 InnoDB 存储引擎中 MVCC 的实现是通过 undo 来完成的；
  4. undo log 的产生会伴随着 redo log 的产生，这是因为 undo log 也需要持久化的保护。

undo、redo log区别

  1. redo log 通常是物理日志，记录的是页的物理修改操作；
  2. undo log 是逻辑日志，根据每行记录进行记录。

五、分布式状态管理

5.1.一些概念

解决方案

mysql读写分离

主服务器在事务执行完成之后会产生一个 bin log 日志，然后bin log 日志传送到从服务器上，从服务器 IO 线程在接收到 bin log 之后会写入到自己的 relay log 中，然后 SQL 线程会从 relay log 中读取数据，并且从服务器会产生一些快照，用于在主服务器数据出错时候的恢复。

分布式事务

资源管理器一般都是指数据库，什么数据库都可以。应用程序首先开始一个事务，然后事务管理器通知资源管理器可以开始了，然后应用程序就会将 SQL 交给资源管理器执行，之后会发送 end 给事务管理器，事务管理器在接收到 end 之后会 询问资源管理器准备好了吗？如果全部执行成功就会 commit，不然就会 rollback。

MySQL内部分布式事务

MySQL 内部分布式事务主要是解决 bin log 和 redo log 不一致的问题。

两阶段提交：InnoDB 首先做一个准备动作，然后 MySQL 写 bin log，然后 InnoDB 写 redo log，如果失败了的话，就可以从 bin log 找回。

7.1.项目实践

安装RocketMQ：

简单使用

在创建生产者的时候需要去给当前生产者取一个名字，然后再通过 producer.setNamesrvAddr(ip:port) 将生产者注册到 Name 服务上，然后再 producer.start() 启动生产者，最后使用的时候需要创建 Topic 以及 tag，并封装为 message，调用 producer.send() 发送消息并重写 SendCallBack() 方法。

消费者那边创建好之后需要 consumer.subscribe(topic,'*') 去订阅主题，然后通过 registerMessageLisener() 方法创建时间监听器。

结合Spring使用

生产者：

生产者通过 MessageBuilder.withPayLoad(message).build 去构建一个消息，然后通过 redisMQTemplate.asyncSend() 去异步的发送消息并重写 SendCallBack() 方法。

消费者：

消费者通过 @RocketMQMessageListener 注解去订阅主题和标签，并声明消费者组，然后通过实现 RocketMQListener() 接口重写 onMessage() 方法去写消费逻辑。

延时消息

下单的时候锁库存可以解决超卖的问题，但是会产生少卖的问题，这个时候就可以用 rocketmq 中的延时消息机制，比如说超过了 15 分钟没有支付就将订单取消。

事务型消息

RocketMQ 的事务型消息要保证本地事务和消费消息的一致性。

事务型消息基本逻辑：

  1. 生产者会发送一个半成品消息给 MQ Server；
  2. MQ Server 在接受到半成品消息的时候会返回给生产者 ok；
  3. 生产者去到 MySQL 执行事务；
  4. 事务执行成功与否生产者都会向 MQ Server 发送一个信号；
  5. 如果第 4 步 中发送给 MQ Server 的信号丢失了就会通知生产者去检查；
  6. 生产者会去重新检测事务成功与否；
  7. 生产者再次向 MQ Server 发送一个信息；
  8. MQ Server 将消息发送给消费者消费。

对应第一步：

项目中的逻辑：

  1. 后端接受到请求后 OrderController.create() 会去调用 orderService.createOrderAsync() 然后通过 itemService.createItemStockLog() 去生成流水信息，接着根据 itemId、amount、itemStockLogId 创建消息，最后通过 rocketMQTemplate.sendMessageInTransaction() 向 MQ Server 发送半成品消息；
  2. MQ Server 返回给生产者 ok；
  3. 自定义的处理消息的 listener 在监听到成功信号后，会去调用 orderService.createOrder() ，然后通过 itemService.findItemInCache() 去缓存中预减库存，执行生成订单的事务，并通过 itemServer.updateItemStockLogStatus() 更新流水状态；
  4. 生成者向 MQ Server 发送完全体消息；
  5. 如果第四步丢失了，MQ Server 就会去检查生产者；
  6. 生产者通过 LocalTransactionListenerImpl.checkLocalTransaction() 去检查流水；
  7. orderService.createOrderAsync() 去向 MQ Server 发送完全体消息；
  8. 消费者去扣减库存。

7.2.热点问题

消息丢失问题

消息可靠性

RocketMQ支持消息的高可靠，影响消息可靠性的几种情况：

  1. Broker非正常关闭
  2. Broker异常Crash
  3. OS Crash
  4. 机器掉电，但是能立即恢复供电情况
  5. 机器无法开机（可能是cpu、主板、内存等关键设备损坏）
  6. 磁盘设备损坏

1)、2)、3)、4) 四种情况都属于硬件资源可立即恢复情况，RocketMQ在这四种情况下能保证消息不丢，或者丢失少量数据（依赖刷盘方式是同步还是异步）。

5)、6)属于单点故障，且无法恢复，一旦发生，在此单点上的消息全部丢失。RocketMQ在这两种情况下，通过异步复制，可保证99%的消息不丢，但是仍然会有极少量的消息可能丢失。通过同步双写技术可以完全避免单点，同步双写势必会影响性能，适合对消息可靠性要求极高的场合，例如与Money相关的应用。注：RocketMQ从3.0版本开始支持同步双写。

重复消费问题

消息重投

生产者在发送消息时，同步消息失败会重投，异步消息有重试，oneway没有任何保证。消息重投保证消息尽可能发送成功、不丢失，但可能会造成消息重复，消息重复在RocketMQ中是无法避免的问题。消息重复在一般情况下不会发生，当出现消息量大、网络抖动，消息重复就会是大概率事件。另外，生产者主动重发、consumer负载变化也会导致重复消息。如下方法可以设置消息重试策略：

  • retryTimesWhenSendFailed:同步发送失败重投次数，默认为2，因此生产者会最多尝试发送retryTimesWhenSendFailed + 1次。不会选择上次失败的broker，尝试向其他broker发送，最大程度保证消息不丢。超过重投次数，抛出异常，由客户端保证消息不丢。当出现RemotingException、MQClientException和部分MQBrokerException时会重投。
  • retryTimesWhenSendAsyncFailed:异步发送失败重试次数，异步重试不会选择其他broker，仅在同一个broker上做重试，不保证消息不丢。
  • retryAnotherBrokerWhenNotStoreOK:消息刷盘（主或备）超时或slave不可用（返回状态非SEND_OK），是否尝试发送到其他broker，默认false。十分重要消息可以开启。

死信队列

死信队列用于处理无法被正常消费的消息。当一条消息初次消费失败，消息队列会自动进行消息重试；达到最大重试次数后，若消费依然失败，则表明消费者在正常情况下无法正确地消费该消息，此时，消息队列 不会立刻将消息丢弃，而是将其发送到该消费者对应的特殊队列中。

RocketMQ将这种正常情况下无法被消费的消息称为死信消息（Dead-Letter Message），将存储死信消息的特殊队列称为死信队列（Dead-Letter Queue）。在RocketMQ中，可以通过使用console控制台对死信队列中的消息进行重发来使得消费者实例再次进行消费。

九、总结

9.1.表

秒杀项目主要涉及7张表，其核心是用户表、商品表、订单表：

  1. 用户表（user_info）
    手机号作为用户名，密码存储的是密文，采用MD5加密。
  2. 商品表（item）
    图片存储的是绝对路径（HTTP），描述存储的是富文本信息（HTML）。
  3. 库存表（item_stock）
    与商品表是多对一关系，将库存从商品表拆分出来，是为了减小修改库存时锁的粒度。
  4. 活动表（promotion）
    与商品表是一对一关系，一个商品只能参与一个活动，我们对秒杀业务模型做了精简的。
  5. 订单表（order_info）
    与商品表是多对一关系，订单中记录了下单时的单价、数量、金额，相当于商品在下单一刻的快照。
  6. 序列表（serial_number）
    用于记录订单等业务场景中的序列号，分别存储了业务名称、当前序号、增长步长。
    生成序号时必须严格互斥，不能采用非锁定读（MVCC），所以我们按订单为维度来生成序号，竞争很大。
    这里可以做进一步优化，比如增加商品列、用户列、服务器列等，将单一的订单维度拆分为更多维度，
    随着维度（粒度）的降低，竞争便会缩小。当然，也可以考虑不用此表生成序列，而是采用雪花算法（百度一下）。
  7. 库存流水表（item_stock_log）
    用户记录商品在下单时的状态流转情况，便于追溯订单的状态，从而为MQ回查订单状态提供依据。
    库存流水表与商品表是多对一关系，记录了商品ID、扣减数量、订单状态。
