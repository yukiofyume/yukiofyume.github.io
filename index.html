可以看到 Transformer 由 Encoder 和 Decoder 两个部分组成，Encoder 和 Decoder 都包含 6 个 block。Transformer 的工作流程大体如下：

第一步：获取输入句子的每一个单词的表示向量 X，X由单词的 Embedding（Embedding就是从原始数据提取出来的Feature） 和单词位置的 Embedding 相加得到。

第三步：将 Encoder 输出的编码信息矩阵 C传递到 Decoder 中，Decoder 依次会根据当前翻译过的单词 1~ i 翻译下一个单词 i+1，如下图所示。在使用的过程中，翻译到单词 i+1 的时候需要通过 Mask (掩盖) 操作遮盖住 i+1 之后的单词。

0、项目架构与运行

0.1.IOC和AOP

IOC

IOC（invert of control） 就是控制反转，是一种思想而不是具体实现，具体来说就是把 bean 的创建交由 Spring 框架进行管理。IOC 容器是 Spring 实现 IOC 的载体，IOC 容器实际上就是一个 Map，里面存放着各种对象。DI（Dependency Injection）依赖注入是 IOC 的一种具体实现。

在我做的项目中 IOC 给我最大的感受就是简单和可拓展性。比如 service 层的下单服务，之后如果对下单服务进行更新和优化，如果没有使用 IOC 就需要将之前 controller 层使用过该服务的代码都需要修改，而如果利用了 IOC 直接新加一个下单服务的 impl，然后再将之前的 impl 的注解掉就行了。

在使用的时候还可以用 @Qualifier("") 显式地将服务写出来，但是不推荐，会使得修改的时候很麻烦。

AOP

AOP（Aspect oriented Programming） 是面向切面编程。项目中在日志模块用到，就是写了一个 LogAspect 类，然后写一个方法加上 @Pointcut("execution(* com.nowcoder.example.service.*.*(..))") 注解声明哪些包下的哪些类哪些方法需要织入，然后再用 @Before @After 等注解写需要去执行的动作。一般都是运行时织入，如果有接口就采用 JDK 动态代理，如果没有接口就采用 CGLIB 动态代理。

0.2.MVC

后端在收到一个请求后会由 DispatcherServlet 统一去处理，然后由 HandlerMapping 会告诉 DispatcherServlet 由哪个 HandlerAdapter 来调用 controller 来处理这个请求，处理完之后会返回给 DispatcherServlet 一个 ModelAndView，然后再由 ViewResolver 处理返回给 DispatcherServlet，最后将处理好的 view 返回给前端。

SSL/TLS协议

发表于 2021-09-09 分类于 计算机基础知识

SSL/TLS 是一种密码通信框架，综合运用了密码学中的对称密码，消息认证码，公钥密码，数字签名，为随机生成器等。

SSL（Secure Socket Layer）安全套接字层；

TLS（Transport Layer Security）传输层安全。

TLS协议的架构

TLS 主要分为两层，底层是 TLS 记录协议，主要负责使用对称密码对消息进行加密；上层是 TLS 握手协议，主要分为握手协议、密码规格变更协议、警告协议、应用数据协议这四个部分。

  • 握手协议：负责在客户端和服务端商定密码算法和共享密钥，包括证书认证；
  • 密码规格变更协议：负责向通信对象传达变更密码方式的信号；
  • 警告协议：负责在发生错误的时候将错误传达给对方；
  • 应用数据协议：负责将 TLS 承载的应用数据传达给通信对象的协议。

手写一个RPC框架

发表于 2021-08-24 更新于 2021-09-09 分类于 RPC学习笔记

0.说明

这是一个基于 Netty 和 Nacos 的 RPC 框架最最最最基本的实现，只是为了学习 RPC 的调用过程以及相关知识

1.实现思路

1.1.RPC介绍

RPC 也就是远程过程调用，生产者服务端向 Ncos 注册中心注册服务，消费者客户端通过注册中心拿到服务相关信息，
然后再通过网络请求向 Server 请求服务。

1.2.需要实现的模块

  1. 注册中心 Nacos：用来负责服务的注册和发现；
  2. 网络传输：需要调用远程方法就需要发送请求，如何确定一个方法呢？需要传输的信息至少要包括
    类名、方法名以及参数列表，这样就可以准确地找到一个方法了。
  3. 序列化：网络传输需要序列化，定义了四种序列化方式，JSON、Hessian、Kyro以及Protobuf。
  4. 动态代理：使用 JDK 动态代理屏蔽远程方法调用的细节，在调用远程方法的时候，通过代理对象来传输网络请求。
  5. 负载均衡：提高可用性。

工作线程组在初始化 channel 的时候设置 hanlder 是自定义的 NettyServerHandler，继承自 SimpleChannelInboundHandler ，重写了 channelRead0() 方法，通过线程池的方式去调用 RequestHandler.handle() ，RequestHandler.handle() 中通过 serviceProvider.getServiceProvider(InterfaceName) 去从服务端本地保存服务的 Map 中取出具体 service，然后通过 invoke() 去调用具体 service，然后将结果交给 ChannelFuture 去异步返回给客户端，然后通过 ChannelFuture.addListner(ChannelFutureListener.CLOSE) 去关闭此 channel。

2.1.客户端使用服务

首先构建一个 NettyClient 实例对象，然后将这个实例对象传入 RpcClientProxy 中，RpcClientProxy 实现了 InvocationHandler 接口去实现服务的 jdk 动态代理，在动态代理中去调用 NettyClient 的 sendRequest() 方法。sendRequest() 首先会去调用 ServiceDiscovery.lookupService() 方法通过负载均衡算法去 Nacos 注册中心找到相应服务所在的地址，接着调用 ChannelProvider.get() 方法去获取 Channel 对象，如果已经存在了存活着的 channel，那么直接返回，不然就重新构建一个。首先创建一个 Bootstrap 的实例对象，参数有 NioEventLoopGroup 事件循环，channel 的类型，超时时间，是否开启 TCP 心跳机制以及设置 TCP 去使用 Nagle 算法去发送尽可能大数据块，减少网络传输，在设置好参数后调用 handler() 构建通信管道，设置序列化反序列化方法，状态处理程序以及自定义的 NettyClientHandler 处理程序。最后使用这个通信管道去处理 channel 的入站以及出站事件，这样一个客户端到 Nacos 注册中心服务发现，以及调用具体服务的 channel 就构建完成了。

客户端在使用的时候首先会创建一个 NettyClient 的实例对象，这个实例对象在创建的时候会在静态代码块中通过 bootstrap 去设置 channels 的基本属性为 Nio 以及长连接，而并没有去创建 channel。同时在使用构造方法的时候会创建一个 serviceDiscovery 的实例对象。

然后使用RpcClientProxy(client) 的 getProxy() 方法去返回动态代理的 helloservice 对象，在调用 helloService.hello() 方法的时候就会去调用动态代理类 RPCClientProxy 的 invoke() 方法，在 invoke() 方法中首先会根据（请求号，待调用接口名，方法名，参数，参数类型）去创建一个 RpcRequest 对象，然后调用 RpcClient.sendRequest(rpcRequest) 去向 Nacos 注册中心发送请求。

请求过程：首先会调用 serviceDiscovery.lookupService(接口名) 去找到相应服务所在的地址，然后调用 ChannelProvider.get() 方法去创建连接到服务端的 channel，如果channel存在就直接使用，不然就通过 Bootstrap 去创建，同时设置 pipeline 上的 handler 为 NettyClientHanlder ，这样 channel 就构建完成了。最后返回 RPCResponse。

Spring常见面试题

发表于 2021-08-24 更新于 2021-11-24 分类于 Spring学习笔记

一、IOC和AOP

1.1.IOC

IOC（invert of control） 就是控制反转，是一种思想而不是具体实现，具体来说就是把对象的创建交由 Spring 框架进行管理。IOC 容器是 Spring 实现 IOC 的载体，IOC 容器实际上就是一个 Map，里面存放着各种对象。DI（Dependency Injection）依赖注入是 IOC 的一种具体实现。

举个例子，我们在使用 dao 层做数据处理的时候可以有多种方式，当我们此时想采用一种新的方式的时候，我们只需要修改相应 dao 接口的实现类，然后将之前实现类的注解去掉不装配，这样就可以在不修改原有代码的基础上去实现新功能，这也就是说为什么 ioc 可以让我们可插拔的去实现我们想要的功能。

bean的生命周期

1.2.AOP

AOP（Aspect oriented Programming） 是面向切面编程，是 OOP（面向对象编程）的一种延续。OOP 通过继承的方式（Animals->(Dog，Cat) 来减少代码冗余，但是如果父类的不同方法中出现了大量的重复性代码（比如日志、性能监控），通过 OOP 是无法消除冗余代码的。这部分重复的代码，叫作横切逻辑代码，横切代码存在代码重复，与业务代码混杂在一起，代码臃肿，不易维护的问题，这个时候就需要 AOP 将业务代码和横切逻辑代码分离出来。

切 ：指的是横切逻辑，原有业务逻辑代码不动，只能操作横切逻辑代码，所以面向横切逻辑。

面 ：横切逻辑代码往往要影响的是很多个方法，每个方法如同一个点，多个点构成一个面。这里有一个面的概念。

原理

  1. 代理的创建
    • 首先，需要创建代理工厂，代理工厂需要 3 个重要信息：拦截器数组、目标对象接口数组、目标对象；
    • 创建代理工厂时，默认会在拦截器数组尾部再增加一个默认拦截器，用于最终调用目标方法；
    • 当调用 getProxy() 方法的时候，会根据接口数大于 0 条件返回一个代理对象（JDK 或 Cglib）；
    • 创建代理对象时，同时会创建一个外层拦截器，这个拦截器就是 Spring 内核的拦截器，用于控制 AOP 的流程。
  2. 代理的调用
    • 当对代理对象进行调用时，就会触发外层拦截器，外层拦截器根据代理配置信息，创建内层拦截器链。创建的过程中，会根据表达式判断当前拦截器是否匹配这个拦截器。而这个拦截器链设计模式就是责任链模式。当整个链条执行到最后时，就会触发创建代理时那个尾部的默认拦截器，从而调用目标方法，最后返回。

SpringSecurity面试题

发表于 2021-08-14 更新于 2021-08-24 分类于 SpringSecurity学习笔记

1.执行流程

  1. 客户端发起一个请求，进入 Security 过滤器链。

  2. 当到 LogoutFilter 的时候判断是否是登出路径，如果是登出路径则到 logoutHandler ，如果登出成功则到 logoutSuccessHandler 登出成功处理，如果登出失败则由 ExceptionTranslationFilter ；如果不是登出路径则直接进入下一个过滤器。

  3. 当到 UsernamePasswordAuthenticationFilter 的时候判断是否为登录路径，如果是，则进入该过滤器进行登录操作，如果登录失败则到 AuthenticationFailureHandler 登录失败处理器处理，如果登录成功则到 AuthenticationSuccessHandler 登录成功处理器处理，如果不是登录请求则不进入该过滤器。

  4. 当到 FilterSecurityInterceptor 的时候会拿到 uri ，根据 uri 去找对应的鉴权管理器，鉴权管理器做鉴权工作，鉴权成功则到 Controller 层否则到 AccessDeniedHandler 鉴权失败处理器处理。

2.登录以及权限

用户认证使用的是 UsernamePasswordAuthenticationFilter 过滤器和 WebSecurityConfigurerAdapter 类。具体来说就是自定义配置类来继承 WebSecurityConfigurerAdapter 类，然后调重写 configure() 方法通过配置 http.addFilterAt() 来对不同用户进行接口的授权。configure() 方法会接收两个参数，第一个参数是一系列的过滤逻辑，比如成功登录了之后怎么做，登录失败了怎么做；然后第二个参数就是 UsernamePasswordAuthenticationFilter.class。

UsernamePasswordAuthenticationFilte：用于处理基于表单的登录请求，从表单中获取用户名和密码。默认情况下处理来自 /login 的请求。从表单中获取用户名和密码时，默认使用的表单 name 值为 username 和 password，这两个值可以通过设置这个过滤器的usernameParameter 和 passwordParameter 两个参数的值进行修改。

3.SpringSecurity核心组件

3.1.SecurityContext

安全上下文，用户通过Spring Security 的校验之后，验证信息存储在SecurityContext中。SecurityContext接口只定义了两个方法，实际上其主要作用就是获取和设置 Authentication 对象。

3.2.SecurityContextHolder

保留系统当前的安全上下文细节，其中就包括当前使用系统的用户的信息。

3.3.Authentication

在Spring Security 中Authentication用来表示当前用户是谁，可以理解为 authentication 就是一组用户名密码信息。Authentication 也是一个接口。

3.4.UserDetails

是一个接口，作用就是存储用户的信息。获取用户权限、获取用户名和密码，账号是否过期、是否被锁定，密码是否过期，账号是否可用。

3.5.UserDetailsService

是一个接口，只有一个 loadUserByUsername() 方法用来获取 UserDetails 信息。

3.6.AuthenticationManager

是一个接口，作用是用来校验 Authentication，如果失败会抛出一个 AuthenticationException 异常。
