SpringSecurity面试题

发表于 2021-08-14 更新于 2021-08-24 分类于 SpringSecurity学习笔记

1.执行流程

  1. 客户端发起一个请求，进入 Security 过滤器链。

  2. 当到 LogoutFilter 的时候判断是否是登出路径，如果是登出路径则到 logoutHandler ，如果登出成功则到 logoutSuccessHandler 登出成功处理，如果登出失败则由 ExceptionTranslationFilter ；如果不是登出路径则直接进入下一个过滤器。

  3. 当到 UsernamePasswordAuthenticationFilter 的时候判断是否为登录路径，如果是，则进入该过滤器进行登录操作，如果登录失败则到 AuthenticationFailureHandler 登录失败处理器处理，如果登录成功则到 AuthenticationSuccessHandler 登录成功处理器处理，如果不是登录请求则不进入该过滤器。

  4. 当到 FilterSecurityInterceptor 的时候会拿到 uri ，根据 uri 去找对应的鉴权管理器，鉴权管理器做鉴权工作，鉴权成功则到 Controller 层否则到 AccessDeniedHandler 鉴权失败处理器处理。

2.登录以及权限

用户认证使用的是 UsernamePasswordAuthenticationFilter 过滤器和 WebSecurityConfigurerAdapter 类。具体来说就是自定义配置类来继承 WebSecurityConfigurerAdapter 类，然后调重写 configure() 方法通过配置 http.addFilterAt() 来对不同用户进行接口的授权。configure() 方法会接收两个参数，第一个参数是一系列的过滤逻辑，比如成功登录了之后怎么做，登录失败了怎么做；然后第二个参数就是 UsernamePasswordAuthenticationFilter.class。

UsernamePasswordAuthenticationFilte：用于处理基于表单的登录请求，从表单中获取用户名和密码。默认情况下处理来自 /login 的请求。从表单中获取用户名和密码时，默认使用的表单 name 值为 username 和 password，这两个值可以通过设置这个过滤器的usernameParameter 和 passwordParameter 两个参数的值进行修改。

3.SpringSecurity核心组件

3.1.SecurityContext

安全上下文，用户通过Spring Security 的校验之后，验证信息存储在SecurityContext中。SecurityContext接口只定义了两个方法，实际上其主要作用就是获取和设置 Authentication 对象。

3.2.SecurityContextHolder

保留系统当前的安全上下文细节，其中就包括当前使用系统的用户的信息。

3.3.Authentication

在Spring Security 中Authentication用来表示当前用户是谁，可以理解为 authentication 就是一组用户名密码信息。Authentication 也是一个接口。

3.4.UserDetails

是一个接口，作用就是存储用户的信息。获取用户权限、获取用户名和密码，账号是否过期、是否被锁定，密码是否过期，账号是否可用。

3.5.UserDetailsService

是一个接口，只有一个 loadUserByUsername() 方法用来获取 UserDetails 信息。

3.6.AuthenticationManager

是一个接口，作用是用来校验 Authentication，如果失败会抛出一个 AuthenticationException 异常。

发表于 2021-07-19 更新于 2021-07-20 分类于 项目实战

一、单点登录

单点登录（Single Sign On），是在多个应用系统中，用户只需要登录一次就可以访问所有相互信任的应用系统。

二、JWT

2.1.RSA非对称加密算法

  1. A要向B发送信息，A和B都要产生一对用于加密和解密的公钥和私钥。

  2. A的私钥保密，A的公钥告诉B；B的私钥保密，B的公钥告诉A。

  3. A要给B发送信息时，A用B的公钥加密信息，因为A知道B的公钥。

  4. A将这个消息发给B（已经用B的公钥加密消息）。

  5. B收到这个消息后，B用自己的私钥解密A的消息。其他所有收到这个报文的人都无法解密，因为只有B才有B的私钥。

三、SpringSecurity整合JWT

3.1.认证思路分析

SpringSecurity 主要是通过过滤器来实现功能的。

集中式认证流程

  • 用户认证：使用 UsernamePasswordAuthenticationFilter 过滤器中 attemptAuthentication() 方法实现认证功能，该过滤器父类中 successfulAuthentication() 方法实现认证成功后的操作。
  • 身份校验：使用 BasicAuthenticationFilter 过滤器中 doFilterInternal 方法验证是否登录，以决定能否进入后续过滤器。

分布式认证流程

  • 用户认证：前后端分离的架构设计，我们要满足可以接受异步 post 的认证请求参数，需要修改UsernamePasswordAuthenticationFilter 过滤器中 attemptAuthentication() 方法，让其能够接收请求体。另外，默认successfulAuthentication() 方法在认证通过后，是把用户信息直接放入 session 就完事了，现在我们需要修改这个方法，在认证通过后生成token并返回给用户。

  • 身份校验：原来 BasicAuthenticationFilter 过滤器中 doFilterInternal() 方法校验用户是否登录，就是看 session 中是否有用户信息，我们要修改为，验证用户携带的 token 是否合法，并解析出用户信息，交给 SpringSecurity，以便于后续的授权功能可以正常使用。

二、消息引擎模型

2.1.点对点模型

点对点模型是基于队列提供消息传输服务的，该模型定义了消息队列、发送者和接收者，提供了一种点对点的消息传递方式，即发送者发送每条消息到队列的指定位置，接收者从指定位置获取消息，一旦消息被消费，就会从队列中移除该消息。每条消息由一个发送者生产出来，且只被一个消费者消费，发送者和消费者之间是一对一的关系。

2.2.发布/订阅模型

发布 / 订阅模型有主题（Topic）的概念。这种模型也定义了类似于生产者 / 消费者这样的角色，即发布者和订阅者，发布者将消息生产出来发送到指定的 topic 中，所有订阅了该 topic 的订阅者都可以接收到该 topic 下的所有信息，通常具有相同订阅 topic 的所有订阅者将接收到同样的消息。

二、策略模式

Spring 在运行时会动态地选择不同的动态代理实现方式。这个场景就是策略模式的典型应用场景。

我们只需要定义一个策略接口，让不同的策略类都实现这一策略接口。

在 Spring 中 AopProxy 是策略接口，JdkDynamicAopProxy、CglibAopProxy 是实现 AopProxy 接口的策略类。

在策略模式中，策略的创建一般通过工厂方法来实现。在 Spring 源码中， AopProxyFactory 是一个工厂类接口，DefaultAopProxyFactory 是一个默认的工厂类，用来创建 AopProxy 对象，选择何种动态代理方式，则由 createAopProxy 方法决定。

策略模式的典型应用场景，一般是通过环境变量、状态值、计算结果等动态地决定使用哪个策略。

三、装饰器模式

缓存一般是配合数据库来使用的。如果写缓存成功，但数据库事务回滚了，那缓存中就会有脏数据。

为了解决这个问题，我们需要将缓存的写操作和数据库的写操作，放到同一个事务中，要么都成功，要么都失败。

实现如上的功能，Spring 使用到了装饰器模式。

TransactionAwareCacheDecorator 增加了对事务的支持，在事务提交、回滚的时候分别对 Cache 的数据进行处理。

TransactionAwareCacheDecorator 实现了 Cache 接口，并且将所以的操作都委托给 targetCache 来实现，对其中的写操作添加了事务功能。这是典型的装饰器模式的应用场景和代码实现。

四、单例模式

单例模式是指一个类在整个系统运行过程中，只允许产生一个实例。

Spring 是通过单例注册表的方式，就是使用了 HashMap 来实现单例模式的。

很明显，从概念上来说，IOU 越大就说明预测的效果越好，越接近真实框。

从几何上来说，影响边界框回归主要有三个因素，即重叠面积、中心点距离和纵横比，在此基础上提出了 CIOU_Loss（Complete IoU） ，从而可以使得模型收敛的更快、取得更好的效果。

IOU 的发展过程：Smooth L1 Loss-> IoU Loss（2016）-> GIoU Loss（2019）-> DIoU Loss（2020）->CIoU Loss（2020）

最基本的 IOU 存在一定的问题，

  1. 当预测框和目标框不相交时，IOU=0，此时损失函数不可导

  2. 当预测框大小相同的时候，两个 IOU 也相同的时候，此时 IOU_loss 无法评估两者相交情况的不同

CIOU 就是为了在预测框和目标框重叠的时候，如何使得模型回归的更加准确。

通过统一坐标，IOU_Loss 考虑了重叠面积问题，DIOU 在考虑了重叠面积以及中心点距离问题，而CIOU_Loss 则同时考虑了重叠面积、中心点距离以及纵横比问题。

这里以无参构造函数为例子，也就是不指定初始容量，也就是说此时初始化赋值了一个空数组，当向数组中添加第一个元素的时候，数组被扩容为10。

先看 add() 方法

在添加元素之前会调用 ensureCapacityInternal() 方法

ensureCapacityInternal():

当 add 第一个元素的时候，经过 calculateCapacity() 方法后会将 DEFAULT_CAPACITY = 10 赋值给 minCapacity，也就是说此时容量变为了 10。

在调用 ensureCapacityInternal() 的时候一定会调用 ensureExplitCapacity() 方法：

  • 当添加第一个元素的时候， len 为 0，因为执行了 ensureCapacityInternal() 方法，此时 minCapacity = 10，因此会调用 grow() 方法。
  • 当添加 2-10 个元素的时候，minCapacity = 2-10，此时 if 不成立，因此不会调用 grow() 方法。
  • 当添加到第 11 个元素的时候， if 成立，进入 grow() 方法进行扩容。
  • 可以看出，当添加第一个元素的时候，满足第 一个 if，所以此时容量设置为了 10；
  • 在扩容的时候，每次都会通过位运算将容量变为 1.5 倍左右；
  • 如果新容量大于了 MAX_ARRAY_SIZE 就会执行，hugeCapacity() 方法来比较 minCapacity（当前元素数量） 和 MAX_ARRAY_SIZE，如果 minCapacity 大于最大容量，那么新容量就是 Integer.MAX_VALUE，不然就是 Integer.MAX_VALUE - 8（将最大容量设置为这样是因为 JVM 在数组对象头中会设置一些参数）。
