设计模式

发表于 2021-05-03 更新于 2021-08-23 分类于 设计模式

一、创建型

1. 单例模式

1.1. 定义

确保一个类只有一个实例，并提供该实例的全局访问点。

使用一个私有构造函数、一个私有静态变量以及一个公有静态函数来实现。私有构造函数保证了不能通过构造函数来创建对象实例，只能通过公有静态函数返回唯一的私有静态变量。

1.2. 实现

  • 懒汉式-线程不安全

    以下实现中，私有静态变量 uniqueInstance 被延迟实例化，这样做的好处是，如果没有用到该类，那么就不会实例化 uniqueInstance，从而节约资源。

    这个实现在多线程环境下是不安全的，如果多个线程能够同时进入 if (uniqueInstance == null) ，并且此时 uniqueInstance 为 null，那么会有多个线程执行 uniqueInstance = new Singleton(); 语句，这将导致实例化多次 uniqueInstance。

  • 双重校验锁-线程安全

    双重校验锁先判断 uniqueInstance 是否已经被实例化，如果没有被实例化，那么才对实例化语句进行加锁。

    第一个 if 语句用来避免 uniqueInstance 已经被实例化之后的加锁操作，而第二个 if 语句进行了加锁，所以只能有一个线程进入，就不会出现 uniqueInstance == null 时两个线程同时进行实例化操作。

    uniqueInstance 采用 volatile 关键字修饰也是很有必要的， uniqueInstance = new Singleton(); 这段代码其实是分为三步执行：

    1. 为 uniqueInstance 分配内存空间
    2. 初始化 uniqueInstance
    3. 将 uniqueInstance 指向分配的内存地址

    但是由于 JVM 具有指令重排的特性，执行顺序有可能变成 1>3>2。指令重排在单线程环境下不会出现问题，但是在多线程环境下会导致一个线程获得还没有初始化的实例。例如，线程 T1 执行了 1 和 3，此时 T2 调用 getUniqueInstance() 后发现 uniqueInstance 不为空，因此返回 uniqueInstance，但此时 uniqueInstance 还未被初始化。

    使用 volatile 可以禁止 JVM 的指令重排，保证在多线程环境下也能正常运行。

pp-yolo

发表于 2021-05-03 更新于 2021-05-13 分类于 深度学习笔记

1. PP-YOLO

1.1. 与 yolov4 的区别

与YOLOv4不同，pp-yolo 没有探索不同的骨干网络和数据扩充方法，也没有使用NAS搜索超参数。 对于主干，直接使用最常见的ResNet作为PP-YOLO的主干。 对于数据扩充，直接使用了最基本的MixUp 。

Mixup数据增强

Mixup数据增强简单来说就是随机的将两张样本按比例进行混合，分类结果也按比例分配，可以增加模型的鲁棒性。比如我们在训练的时候，猫对应的标签就是猫，狗就是狗，但是，如果实际情况中，出现了一个既像猫又像狗的样本，但是实际上这个样本为狗，此时网络就没有办法对这个样本有效的分类了。Mixup数据增强降低了ResNet系列网络的 TOP-1 ERROR 0.2-0.3 个百分点。

Top-1 ERROR:假设有十个分类，现在有十张测试图片，那么一张图片输入到网络中，就可以得到十个分类概率。Top-1是这十个类别概率中取概率最高的一个类别，如果此测试图片的类别为概率最高类别，则表明预测成功，反正预测错误。

本文为什么不去改变骨干网络，原因之一是ResNet被更广泛地使用，从而使各种深度学习框架针对ResNet系列进行了深度优化，这将在实际部署中更加方便，并且在实践中将具有更快的推断速度。 另一个原因是骨干网络的替换和数据扩充是相对独立的因素。使用NAS搜索超参数通常会消耗更多的计算能力，因此通常没有条件在每个新方案中使用NAS来执行超参数搜索。

使用更好的骨干网络，使用更有效的数据扩充方法以及使用NAS搜索超参数可以进一步提高PP-YOLO的性能。（文中的引言就指出了可以提升的地方，相当于为 PP-YOLOv2 留了个白）

主页展示有试卷总数、题目总数、答卷总数、答题总数。前端在发送一个请求后通过 webpack proxy 转发到后端 DashboardController 中对应的处理方法 index() ，通过 service 层去调用 dao 层的相应方法去数据库 exam_paper、question、exam_paper_answer、exam_paper_question_customer_answer 中查询出所需要的试卷总数、题目总数、答卷总数、答题总数。

学生列表

学生列表会展示id、用户名、真实姓名、年级、性别、手机号、创建时间、状态、操作信息。前端发送一个请求转发到 UserController 之后通过 service 层去调用 dao 层的相关方法根据 role 字段去数据库中查询，如果 role = 1 就说明是学生，返回相应信息后通过 PageHelper 分页返回到前端并展示。

2.3.卷题管理

题目列表

题目列表主要展示id、学科、题型、题干、分数、难度、创建时间以及操作 。前端发送请求后会对应后端的 EducationController 和 QuestionController 。QuestionController 中的 pageList() 方法首先通过 questionService 调用 dao 层根据 VM 中的条件查询 question 表中数据返回给 PageInfo，然后在根据查询出来的 info_text_conten_id 去调用 textContentService 查询出题干。EducationController 通过 subjectService 调用 dao 层查询 subject 表然后返回信息。这样完成了展示数据的查询。

QuestionPageRequestVM：id、level、subjectId、questionType。ExamPaperController 中的 pageList() 方法首先通过 examPageService 调用 dao 层根据 VM 去条件查询 exam_paper 表。

试卷列表

试卷列表需要展示id、学科、试卷名称、创建时间。前端发送请求后会对应后端的 ExamPaperController ，然后根据 mapping 找到对应的方法，然后根据传递过来的 vm 去 service 层调用 dao 层去数据库中查询相关信息。同时会有请求到达 EducationController ，通过 service 层调 dao 层然后去数据库中查询出所有学科信息，返回给前端，由前端根据 subject_id 去完成拼接。

2.4.任务管理

创建任务

前端发送请求后会对应后端 ExamPaperController ，service 层根据前端传递的 vm 参数去调用 dao 层对数据库进行查询，然后将结果返回。

任务列表

前端需要展示的有id、标题、年级、发送者、创建时间。对应后端 TaskController ，service 层根据前端传递的 vm 参数去调用 dao 层去数据库查询，然后将结果返回。

2.5.教育管理

2.6.成绩管理

前端发送请求后对应后端的 ExamPaperAnswerController 和 EducationController。展示的数据有id、试卷名称、用户名称、得分、题目对错、耗时、提交时间。ExamPaperAnswerController 会根据学科id（可以为空）进行条件查询 exam_paper_answer、subject，通过 exam_paper_answer 查询出来的 create_user 字段去 user 表中查询相应的用户名称，最后返还给前端，这样就完成了答卷列表的查询与展示。

2.7.消息中心

消息列表

前端发送请求后对应后端的 MessageController 。展示的数据有id、标题、内容、发送人、接收人、已读数、接收人数、创建时间。pageList() 方法会根据 sendUserName（可以为空） 首先去查询 message表，然后根据 message_id 去 message_user 中查询出接收人、已读数以及接收人数，然后将这些数据封装为 vm 返回给前端。

消息发送

前端发送请求后对应后端的 UserController。展示的数据有id、用户名、真实姓名、动态、创建时间。根据用户id、用户名（都可以为null）从 user_event_log 中查询数据并返回给前端。

Java基础

发表于 2021-04-30 更新于 2021-08-24 分类于 Java学习笔记

一、 基础

1.1. 谈谈final、finally、finalize的区别

  • final：

    • final 类不能被继承，没有子类，final 类中的方法默认是 final 的；
    • final 方法不能被子类覆盖，但是可以被继承；
    • final 成员变量表示的是常量，只能被赋值一次，赋值后不再改变；
    • final 不能用于构造方法。

  • finally

    是异常处理后的一个代码块，无论是否抛出异常，都会执行。

  • finalize：（这个方法建议别使用）

    finalize 是方法名，是垃圾收集过程中对象可能调用的方法。任何一个对象的finalize()方法都只会被系统自动调用一次，如果对象面临下一次回收，它的finalize()方法不会执行。

    在可达性分析算法中判定为不可达的对象，并不会被直接清理掉，要真正宣告一个对象死亡，最多会经历两次标记过程：

    • 如果对象在进行可达性分析后发现没有与 GC Roots 相连接的引用链，那它将会被第一次标记，随后进行一次筛选，筛选的条件是此对象是否有必要执行 finalize() 方法。假设对象没有覆盖 finalize() 方法，或者 finalize() 方法已经被虚拟机调用过，那么虚拟机将这两种情况都视为 没有必要执行
    • 如果这个对象被判定为有必要执行 finalize() 方法，那么该对象将会被放置在一个名为 F-Queue 的队列中，并在稍后由一条虚拟机自动创建、低调度优先级的 Finalizer 线程去执行它们的 finalize() 方法。这里所说 “执行” 是指虚拟机会触发这个方法开始运行，但并不承诺一定会等待它运行结束。这样做的原因是，如果某个对象的 finalize() 方法执行缓慢，或者更极端地发生了死循环，将很可能导致 F-Queue 队列中的其它对象永远处于等待，甚至导致整个内存回收子系统的崩溃。**finalize()** 方法是对象逃脱死亡命运的最后一次机会，稍后收集器将对 F-Queue 中的对象进行第二次小规模的标记，如果对象要在 finalize() 中成功拯救自己，只要重新与引用链上的任何一个对象关联即可，譬如把自己（this 关键字）赋值给某个类变量或者对象的成员变量，那么第二次标记时它会被移出 “即将回收” 的集合；如果这个时候对象还没有逃脱，那基本上它就真的要被回收了。

1.2.抽象类和接口

抽象类

抽象类和抽象方法都使用 abstract 关键字进行声明。如果一个类中包含抽象方法，那么这个类必须声明为抽象类。

接口

jdk8 开始，接口可以有拥有默认的方法实现，这是因为不支持默认方法的接口的维护成本太高了，jdk8 之前，接口添加一个方法后，就需要修改实现了该接口的所有类。

接口的成员默认是 public 的，不允许定义为 private 或者 protected。从 jdk9 开始，允许将方法定义为 private，这样就能定义某些复用的代码又不会把方法暴露出去。

比较

  1. 从设计层面上看，抽象类提供了一种 IS-A 关系，需要满足里式替换原则，即子类对象必须能够替换掉所有父类对象。而接口更像是一种 LIKE-A 关系，它只是提供一种方法实现契约，并不要求接口和实现接口的类具有 IS-A 关系。
  2. 从使用上来看，一个类可以实现多个接口，但是不能继承多个抽象类。
  3. 接口的字段只能是 static 和 final 类型的，而抽象类的字段没有这种限制。
  4. 接口的成员只能是 public 的，而抽象类的成员可以有多种访问权限。

二、代理模式

2.1. 静态代理

静态代理实现步骤:

  1. 定义一个接口及其实现类；
  2. 创建一个代理类同样实现这个接口
  3. 将目标对象注入进代理类，然后在代理类的对应方法调用目标类中的对应方法。这样的话，我们就可以通过代理类屏蔽对目标对象的访问，并且可以在目标方法执行前后做一些自己想做的事情。

2.2. 动态代理

JDK 动态代理

JDK 动态代理步骤

  1. 定义一个接口及其实现类；
  2. 自定义 InvocationHandler 并重写invoke()方法，在 invoke() 方法中我们会调用原生方法（被代理类的方法）并自定义一些处理逻辑；
  3. 通过 Proxy.newProxyInstance(ClassLoader loader,Class<?>[] interfaces,InvocationHandler h) 方法创建代理对象；

具体来说

  • 首先定义一个接口 SmsService 以及实现类 SmsServiceImpl ；

  • 新建类 DebugInvocationHandler 来实现 InvocationHandler 接口并重写 invoke() 方法来实现自己的逻辑；

  • 定义 JdkProxyFactory 类中定义具体的代理实例；

  • 在自己使用的地方调用工厂中的方法即可。

CGLIB 动态代理机制

介绍

JDK 动态代理有一个最致命的问题是其只能代理实现了接口的类。

CGLIB(Code Generation Library)是一个基于的字节码生成库，它允许我们在运行时对字节码进行修改和动态生成。CGLIB 通过继承方式实现代理。很多知名的开源框架都使用到了， 例如 Spring 中的 AOP 模块中：如果目标对象实现了接口，则默认采用 JDK 动态代理，否则采用 CGLIB 动态代理。

代理步骤

  1. 定义一个类；
  2. 自定义 MethodInterceptor 并重写 intercept() 方法，intercept() 用于拦截增强被代理类的方法，和 JDK 动态代理中的 invoke() 方法类似；
  3. 通过 Enhancer 类的 create()创建代理类；

具体示例

  • 定义一个类 SmsService
  • 定义一个方法拦截器 DebugMethodInterceptor 实现 MethodInterceptor 接口，重写 intercept() 方法来增加代理类的方法
  • 定义 CglibProxyFactory 类，在其中实现 Enhancer 类的 create() 方法来创建代理类。

JDK 动态代理和 CGLIB 动态代理对比

  1. JDK 动态代理只能代理实现了接口的类或者直接代理接口，而 CGLIB 可以代理未实现任何接口的类。 另外， CGLIB 动态代理是通过生成一个被代理类的子类来拦截被代理类的方法调用，因此不能代理声明为 final 类型的类和方法；
  2. 就二者的效率来说，大部分情况都是 JDK 动态代理更优秀，随着 JDK 版本的升级，这个优势更加明显。

2.3. 静态代理和动态代理的对比

  1. 灵活性 ：动态代理更加灵活，不需要必须实现接口，可以直接代理实现类，并且可以不需要针对每个目标类都创建一个代理类。另外，静态代理中，接口一旦新增加方法，目标对象和代理对象都要进行修改，这是非常麻烦的！
  2. JVM 层面 ：静态代理在编译时就将接口、实现类、代理类这些都变成了一个个实际的 class 文件。而动态代理是在运行时动态生成类字节码，并加载到 JVM 中的。

三、Java 的三种 I/O 模式

3.1. BIO

Blocking I/O 属于同步阻塞 I/O 模型，同步阻塞 I/O 模型中，在应用程序发起 read 调用后，会一直阻塞，直到内核完成拷贝数据

3.2. NIO

Non-blocking I/O 可以看作是 I/O 多路复用模型

I/O 多路复用模型中，线程首先发起 select 调用，询问内核数据是否准备就绪，等内核把数据准备好了，用户线程在发起 read 调用，read 调用的过程（数据从内核空间 -> 用户空间）还是阻塞的。

3.3. AIO

Asynchronous I/O 是异步 I/O ，异步 I/O 是基于事件和回调机制实现的，也就是应用操作之后会直接返回，不会阻塞在那里，当后台处理完成，操作系统会通知相应的线程进行后续的操作。

通俗来说，注解类似于标签。

4.1.注解的使用

4.2.元注解

元注解就是注解上的注解，有五种。

@Retention

当 @Retention 应用到一个注解上的时候，它解释说明了这个注解的的存活时间。

  • RetentionPolicy.SOURCE 注解只在源码阶段保留，在编译器进行编译时它将被丢弃忽视。
  • RetentionPolicy.CLASS 注解只被保留到编译进行的时候，它并不会被加载到 JVM 中。
  • RetentionPolicy.RUNTIME 注解可以保留到程序运行的时候，它会被加载进入到 JVM 中，所以在程序运行时可以获取到它们。

@Documented

顾名思义，这个元注解肯定是和文档有关。它的作用是能够将注解中的元素包含到 Javadoc 中去。

@Target

Target 是目标的意思，@Target 指定了注解运用的地方。可以这样理解，当一个注解被 @Target 注解时，这个注解就被限定了运用的场景。

  • ElementType.ANNOTATION_TYPE 可以给一个注解进行注解
  • ElementType.CONSTRUCTOR 可以给构造方法进行注解
  • ElementType.FIELD 可以给属性进行注解
  • ElementType.LOCAL_VARIABLE 可以给局部变量进行注解
  • ElementType.METHOD 可以给方法进行注解
  • ElementType.PACKAGE 可以给一个包进行注解
  • ElementType.PARAMETER 可以给一个方法内的参数进行注解
  • ElementType.TYPE 可以给一个类型进行注解，比如类、接口、枚举

@Inherited

Inherited 是继承的意思，但是它并不是说注解本身可以继承，而是说如果一个超类被 @Inherited 注解过的注解进行注解的话，那么如果它的子类没有被任何注解应用的话，那么这个子类就继承了超类的注解。

4.5.注解的使用场景

注解是一系列元数据（数据的数据），它提供数据用来解释程序代码，但是注解并非是所解释的代码本身的一部分。注解对于代码的运行效果没有直接影响。

注解有许多用处，主要如下：

  • 提供信息给编译器： 编译器可以利用注解来探测错误和警告信息。
  • 编译阶段时的处理： 软件工具可以用来利用注解信息来生成代码、Html文档或者做其它相应处理。
  • 运行时的处理： 某些注解可以在程序运行的时候接受代码的提取。
    值得注意的是，注解不是代码本身的一部分。

当开发者使用了Annotation 修饰了类、方法、Field 等成员之后，这些 Annotation 不会自己生效，必须由开发者提供相应的代码来提取并处理 Annotation 信息。这些处理提取和处理 Annotation 的代码统称为 APT（Annotation Processing Tool)。
